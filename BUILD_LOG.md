# Sleep Light Study — 搭建日志

本文记录网站从需求定义、实验实现到公开部署的主要过程。时区为 Asia/Shanghai。第 1–14 节保留 v1/v2 当时的实现历史；第 15–20 节记录首个 2026-07-18 Protocol v3；第 21 节起记录双语姓名档案版本。旧版“Participant ID”“20 次十字”“不能过夜恢复”等描述是当时的历史状态，不代表新版本会改写或删除那些记录。

## 1. 初始需求

目标是搭建一个睡前光照研究网页，包含：

- 约五分钟的纯色视觉刺激
- 亮红、暗红、亮蓝、暗蓝四种条件
- 实验过程中随机出现黑色十字
- 参与者点击屏幕或按键证明持续注意
- 记录十字出现时间和参与者响应时间

## 2. 界面与研究流程

采用深色沉浸式研究界面作为开始页，实验刺激页保持全屏纯色，避免无关界面元素影响刺激。

v2 阶段流程（截至 2026-07-11）：

1. 输入 Participant ID
2. 选择指定条件
3. 阅读英文任务说明并确认开始
4. 三秒倒计时
5. 五分钟有效实验时间的全屏刺激与注意力任务
6. 结果统计与 CSV/JSON 导出

英文说明会在实验开始前明确告知参与者保持专注、等待黑色十字，并在十字出现时按 `Space` 或点击屏幕；同时说明 `P` 暂停/继续和依次输入 `E`、`N`、`D` 提前终止的控制方式。

## 3. 实验引擎实现

初版使用连续随机延迟安排十字。代码审查发现，如果下一次十字从参与者响应后才开始计时，反应较快的人会得到更多试次，影响条件可比性。

v2 随后改为：

- 每次会话开始时预先生成固定 `20` 次试次（该设计在 v3 中被四次稀疏十字取代，但历史 v2 数据保持不变）
- 将 290 秒可用区间划分为 20 个 14.5 秒时间窗，在每个窗口的 25%–75% 区间随机选择计划出现时间
- 所有计划与参与者的点击速度独立
- 十字坐标也在会话开始时预先生成并写入计划

## 4. 计时精度改进

初版在 React 更新界面前记录出现时间，可能让记录时间早于真实绘制时间。

改进后：

- 先同步提交十字界面
- 再在浏览器下一次 `requestAnimationFrame` 中记录视觉 onset
- 保存计划 onset 和实际 onset
- 反应时使用实际 onset 计算
- 持久化操作延后到绘制之后，减少对显示时机的影响

## 5. 注意力与环境记录

除命中和漏答外，增加了以下记录：

- 无目标时的误点
- 页面隐藏和恢复
- 进入和退出全屏
- 全屏请求失败
- 指针坐标、十字坐标和输入方式

如果浏览器隐藏或计时器被节流，已经错过显示窗口的试次不会连续补播，而会标记为 `omitted`。v2 正常完成时会补全所有未能呈现的计划试次，确保完整会话保留 20 次结果；使用 `END` 提前终止时，未来试次只保留在 `trialPlan`，不会被误算为漏答或 `omitted`，正在进行的试次会标记为 `cancelled`。

随后根据实验控制需求增加：

- 在运行时按 `P` 暂停，再按 `P` 继续
- 暂停期间停止刺激和注意力试次，并冻结五分钟有效实验倒计时
- 记录每次暂停的开始时间、结束时间和持续时间
- 在运行或暂停时依次输入 `E`、`N`、`D` 提前终止，并将会话状态记录为 `terminated`
- 正常完成的会话状态记录为 `completed`
- 刺激页左下角显示小型剩余时间倒计时
- 无十字时的点击，以及一次命中后的额外点击，统一记录为 `false_click`

## 6. 刺激可见性处理

审查发现，纯黑十字在 `#660000`、`#0000FF` 和尤其 `#000066` 上的视觉对比较低。

当前处理是保留黑色十字主体，同时增加细浅色轮廓。这样可以提高暗色条件下的可见性，但正式研究仍需要在实际设备上进行对比度与物理亮度校准。

## 7. v2 数据结构与导出

实现了：

- 会话级 JSON 记录
- 事件级 CSV 导出
- 本地浏览器自动保存
- 最终记录自动上传到 Supabase，并保留本地失败重试副本
- 新版重试队列使用 `sleep-light-study:sessions:v2`；旧版 `v1` 非 UUID 记录不会被重复提交或删除
- 通过 `admin` 保留值进入密码保护的远程数据页面
- 管理员远程搜索、分批加载全部有效记录、刷新和下载全部或单次 CSV/JSON
- CSV 事件按时间排序
- CSV 始终写入 `session_summary`，确保开始后立即 `END` 也保留完整会话元数据
- 参与者编号的电子表格公式注入防护
- 实验结束后的命中数、漏答数、平均反应时和准确率统计
- 暂停事件的开始、结束和持续时间记录
- 实际有效实验时长、墙钟总时长和暂停总时长分别记录
- 正常完成与提前终止的会话状态记录
- `false_click` 事件记录，包括无十字点击和命中后的多余点击

平均反应时只使用 `hit` 试次；准确率为 `hit / (hit + missed)`，不包含 `omitted` 与误点。

暂停不会减少预设的五分钟有效实验时间；继续后，计划试次和剩余倒计时从暂停点恢复。CSV 使用独立的暂停事件行保存暂停开始、结束和时长，JSON 保留完整会话级暂停记录。

远程端只接收 `completed` 或 `terminated` 最终记录；频繁产生的 `active` 快照仍只保存在本地。最终记录远程保存成功后会从本地重试队列清除，失败记录会在下次打开页面时再次提交。`test` 和 `admin` 均被客户端与数据库约束排除。数据库开启并强制 RLS：匿名用户只有新增权限，管理员读取权限通过 Supabase Auth 用户 UUID 的私有 allow-list 判断。

匿名写入能保护记录不被参与者读取或修改，但不能证明每条提交一定来自真实实验页面。正式研究如果需要更强的数据完整性，应升级为 Supabase Edge Function，并加入一次性 participant token、速率限制或其他防滥用措施。

## 12. v2 阶段的研究解释限制

- 浏览器无法记录实际 lux、光谱功率分布、环境光或显示器校准状态。
- 当前版本不记录设备型号、浏览器版本、刷新率或窗口尺寸。
- 条件由操作员手选，没有内置随机分组、条件平衡、盲法或重复 Participant ID 检查。
- 刷新、关闭、休眠后不会续跑正在进行的实验。
- `localStorage` 可能被用户清理或在隐私模式下受限，完成后必须及时下载。
- 匿名数据库新增接口可被伪造或滥用；RLS 保护记录机密性，但不能独立证明研究真实性。
- 本任务只记录短时视觉注意反应，不记录入睡时间、睡眠质量或其他睡眠结局。

## 13. 内置 Test mode

根据试用需求，加入保留 Participant ID `test`（不区分大小写）：

- 可反复运行完整五分钟实验
- 所有计时、十字和结果统计仍正常工作
- 暂停/继续、`END` 提前终止和左下角倒计时仍正常工作
- `persist()` 在检测到 test participant 后直接返回内存记录，不写入 `localStorage`
- 历史会话导出会过滤 test participant
- 结果页隐藏 CSV/JSON 下载，并明确说明记录未保存
- 进入页面时会清理旧版本可能留下的 test participant 本地记录
- Participant ID 输入框不公开提示 `test`，避免参与者得知研究者测试入口

## 14. 触摸设备控制

为解决手机和 iPad 不便调出键盘的问题，网页现在检测输入能力而不是解析设备名称：`navigator.maxTouchPoints`、`(pointer: coarse)` 和真实 `pointerType="touch"` 任一满足，就启用触摸模式。这样即使 iPad 使用桌面浏览器标识，也能显示正确界面；带触摸屏的电脑显示额外按钮属于安全降级，键盘快捷键仍然保留。

触摸模式的英文教程只说明点击屏幕和底部控件。实验期间显示 Pause/Resume 与 End；End 第一次点击只进入三秒确认状态，第二次点击才终止，终止原因保存为 `touch_end`。控制栏与整屏反应区域分离，并阻止 pointer 事件冒泡，因此操作按钮不会被误记为命中或 `false_click`。所有设备的十字纵向位置统一限制在 15%–60%，包括较矮的横屏设备，并保持各设备使用相同的位置分布。

教程页同时提供低调的手动模式切换，解决触屏笔记本等混合设备可能更适合键盘操作的情况；参与者手动选择后，后续触摸事件不会覆盖该选择。暂停、恢复、结束和阶段切换都会清除未完成的 `E N D` 序列，避免跨暂停继续旧指令。

## 15. Protocol v3：降低十字频率

研究设计复核后认为，五分钟内出现 20 次目标可能使参与者持续保持过高警觉，反而改变睡前状态。v3 因此新增 `attentionProtocolVersion: "sparse-4-50-70-v1"`：

- 每个颜色条件预生成固定 `4` 次黑色十字，不超过四次
- 相邻计划 onset 间隔为 `50–70` 秒
- 十字仍显示 `1,800 ms`，并继续记录计划 onset、实际绘制 onset、点击和误点
- 四次计划不受参与者回应速度影响
- v2 的 20 次历史记录不迁移、不重写，数据库同时接受 schema 2 和 schema 3

Control 没有颜色曝光，因此 `trialPlan` 和 `trials` 都严格为空。分析时必须按 `schemaVersion` 和 `attentionProtocolVersion` 区分旧版高频任务与新版稀疏任务。

## 16. 第五个条件：Control — Normal Sleep

v3 把 Control 作为与四种颜色并列的第五个正式条件。它表示整晚按照平常方式睡眠，而不是黑色、暗色或“零亮度视频”：

- 不进入五分钟屏幕颜色曝光
- 不改变网页亮度
- 不呈现十字注意力任务
- 仍完成睡前问卷、睡前 KSS、整夜等待、睡醒后 KSS 和反应检查

Control 会话保存 `plannedDurationMs = 0`、`exposureStatus = "not-applicable"` 和空注意力数组。网页不强制 washout day；多个条件之间是否间隔、间隔多久以及条件顺序由研究者的预注册方案决定。

## 17. 睡前/睡醒问卷与设备记录

所有五种条件在睡前使用同一份版本化问卷。问卷包括：

- 最近一次睡眠时尝试入睡的时间
- 完整标注版 KSS 1–9（回顾刚刚过去五分钟）
- 会话前两小时是否使用电子屏幕以及估计分钟数
- 今晚是否开灯睡觉以及灯光颜色类别
- 睡眠环境主观温度
- 是否使用助眠药物或保健品；只记录 Yes / No / Prefer not to answer，不收集药名
- 最近醒来时的休息恢复感 1–5 和最近睡眠质量 1–5
- 过去八小时咖啡因、入睡音乐、环境噪音等级和过去十二小时剧烈运动

行为和环境项目作为独立协变量保存，不合成为未经验证的总分。睡醒后再次使用同一 KSS 1–9，并特意放在反应测试说明之前。KSS 的 1–9 英文标签完整保存在 `app/protocol-v3.ts`，问卷分别带有 `pre-study-v1` 与 `post-study-v1` 版本标识。

网页不保存 User-Agent。它根据触点数量、粗/细指针、悬停能力和屏幕短边自动分类 Phone、Tablet 或 Computer，参与者可以修正。睡前和睡醒后各保存一次设备信息，并记录 `deviceChanged`，供分析设备变化造成的计时差异。

## 18. 睡醒后的三次反应检查

睡醒后 KSS 完成后，参与者会先看到“放松并自然回应”的说明。反应流程包括一次练习，并继续到取得三次有效正式回应：

- 每次目标等待时间随机为 `2–5` 秒
- 目标出现后的回应窗口为 `2` 秒
- 提前回应记为 `false-start`
- 超时记为 `missed`
- 提前回应和漏答分别累计计数，然后重做当前编号，不会被当成有效回应
- 最终只使用三次 `valid` 回应计算平均值和中位数；同时保存 `false-start` 与 `missed` 累计数量

该模块标识为 `relaxed-reaction-test-v1`，属于简短浏览器反应检查，不应描述成完整或临床验证的 PVT。绝对毫秒值仍会受到设备与输入方式延迟影响。

## 19. 48 小时受保护草稿与恢复

v3 跨越一整晚，不能再沿用 v2 “最终结束后才上传”的单阶段假设。新增过夜草稿协议 `overnight-v1`：

- 浏览器生成 32 字节随机恢复令牌并持有其 64 位十六进制原文
- Supabase 使用 PostgreSQL 核心 SHA-256 函数保存令牌摘要，不依赖 `pgcrypto`
- 草稿存于 `private.study_drafts`，匿名和普通已登录角色均无表级权限
- 匿名页面只能执行三个最小权限 RPC：保存、按令牌读取和删除自己的草稿
- 草稿必须符合 active `overnight-v1` schema v3 结构且不超过 `128 KiB`
- 睡前问卷后保存 `not-started` 草稿；颜色阶段使用 `in-progress` 检查点记录试次、回应、暂停和环境事件，意外刷新则保存为 `page_reload` 提前中断
- 每次保存后有效期更新为 48 小时；读取会拒绝并清理已过期草稿
- 睡眠标记、睡醒返回和睡醒后问卷都必须先成功写入同一浏览器，页面才进入下一阶段；正式会话另设四小时防误点门槛
- 完成睡醒后问卷和反应检查、成功形成最终记录后删除草稿

恢复能力依赖参与者仍持有 bearer token；Participant ID 本身不能用于查询草稿。该保护机制不是客户端端到端加密，文档和同意材料不能把它描述成“加密问卷”。

## 20. Supabase v3 兼容迁移

全新项目可运行更新后的 `supabase/setup.sql`。已经运行 v2 的项目使用独立迁移文件 `supabase/migrations/20260718_protocol_v3.sql`：

- 保留所有 v2 行和原有管理员 allow-list
- 为最终会话增加 `control` 和 schema v3 约束
- v3 颜色条件要求四次计划，Control 要求零次计划
- v3 最终 payload 要求设备、前测、后测和反应检查对象
- 创建私有草稿表、PostgreSQL 核心 SHA-256 token 支持和三个受限 RPC

迁移成功时 SQL Editor 正常显示 `Success. No rows returned`。上线顺序应是：恢复/确认 Supabase 项目健康、运行迁移、完成端到端测试、再发布 v3 网页；否则新网页提交会被旧数据库的四条件/20 次约束拒绝。

## 21. 中英双语界面与完整参与者教程

新版本在首页加入 English / 中文切换，并让所选语言贯穿姓名档案、设备确认、教程、睡前问卷、颜色/Control 阶段、睡醒流程、KSS、反应检查、结果和反馈。教程不只说明按键，还覆盖整个实验过程：

1. 每次实验尽量使用相同设备和显示设置，并保持相近的温度、计划入睡时间、声音/噪音、灯光、睡前屏幕使用、音乐、咖啡因和助眠品习惯。
2. 一致性要求不能取代诚实回答；参与者不应为了“答案相同”而报告虚假环境，也不应为了研究留在不安全或不舒适的环境。
3. 颜色条件中保持注视，十字出现时点击/轻触或按 `Space` / `Enter`；电脑端保留 `P` 和 `E N D`，触摸端使用底部控制。
4. Control 跳过颜色、亮度和十字阶段，按平常方式完成整晚睡眠。
5. 睡醒后返回同一网页，完成第二次 KSS、一次练习和三次有效自然反应，再选择提交或跳过反馈。

英文教程表达同样的研究要求，不是中文内容的简化版。两种语言切换只改变呈现语言，不改变条件、计时、问卷字段或记录结构。

## 22. 唯一实验姓名、隐私和恢复码

开始页的 `Participant ID` 改为 `Study name / 实验姓名`。参与者可以使用真名或网名；为减少不必要的身份暴露，界面明确推荐使用不会直接识别本人、也不包含邮箱或学校编号的网名。

- 姓名经 NFKC 规范化、首尾清理、内部 Unicode 空白合并和大小写不敏感比较后保持唯一。
- 首次建立姓名档案时生成 20 位恢复码；原始恢复码保留在参与者浏览器或由参与者自行私下保存。
- 浏览器把恢复码转换为 SHA-256 proof 后发送；数据库再哈希该 proof，不保存可显示的原始恢复码。
- 返回者可使用原浏览器，或在其他设备输入相同姓名和恢复码，重新打开同一个档案。
- 恢复码无法由管理员反向取回，参与者不应分享；它用于证明对该姓名历史的访问权，而不是研究者可重置的普通密码。

唯一姓名不能保证其一定是真实身份，也不应被当作身份验证。它的目的只是让每个人的多晚记录整洁地归入同一研究档案。

## 23. 五条件历史与只追加反馈

姓名档案显示 Bright Red、Dim Red、Bright Blue、Dim Blue 和 Control 的完成/剩余状态，并保留每次完成的会话历史。网页只汇总进度，不自动分配下一个条件、不随机化顺序，也不阻止连续晚实验；顺序继续由研究方案决定。重复完成同一条件会新增一条会话，不会覆盖前一次。

每次完整实验后显示可选的 Feedback / Question 窗口：

- 参与者可选择反馈或问题、输入最多 4,000 字符并提交，也可以跳过。
- 每次提交都生成独立 ID，并保存时间、界面语言、提示版本和网站构建版本。
- 反馈是 append-only；新的留言不会更新或替换旧留言。
- 管理员可以按姓名和会话查看所有历史留言，但普通参与者不能枚举其他人的留言。

## 24. 管理员黄色环境一致性提醒

管理员仪表板按唯一姓名汇总已完成的 v3 会话，并在以下任一条件成立时显示黄色感叹号，提示研究者人工复核：

- 用 24 小时圆周算法处理跨午夜后，报告的入睡时间最小跨度超过 90 分钟；
- 主观温度从最低到最高相差超过一个有序等级；
- 环境噪音从最低到最高相差超过一个有序等级；
- 是否开灯睡觉，或开灯时的灯光颜色发生变化；
- 屏幕使用、入睡音乐、过去八小时咖啡因、助眠品四项中至少两项出现 Yes/No 变化。

`Prefer not to answer` 不用于制造变化。研究者分配的颜色/Control 条件变化是实验自变量，**永远不会**触发环境不一致提醒。黄色感叹号只表示“需要认真查看”，不会自动排除、删除或更改会话。

## 25. 构建版本和历史记录保留

本版本为每个新会话写入 `studyBuildVersion`；反馈另保存提示版本和构建版本。这样导出和管理员页面可以把每份答案追溯到参与者实际使用的网站构建，而不是假设所有 v3 记录都来自相同界面。

历史策略为只追加与增量升级：

- 旧 schema v2、早期 schema v3、每次问卷答案、条件结果和反馈都保留原值；
- 新版本不会把旧答案转换成新字段，不会用同姓名的新会话覆盖旧会话，也不会自动删除旧记录；
- 新迁移为现有表增加档案关联、反馈和 append-only 保护，而不是重建或清空 `study_sessions`；
- 不同问卷、提示、协议和网页构建必须依靠其版本字段分层分析。

No rows returned`。迁移前后 `study_sessions` 都是 0 条；三张档案/反馈表、七个公开 RPC、六条 v3 约束、三个草稿 RPC、只追加触发器和管理员 allow-list 随后全部通过核对。匹配的网页构建发布到 GitHub Pages，GitHub Actions #40 成功，英文、中文、五条件和不保存数据的 `test` 教程流程通过线上冒烟测试。
